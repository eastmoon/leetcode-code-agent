# 程式碼摘要憲章 (Code Summarization Constitution)

## Core Principles

### I. 意圖提升 (Intent Elevation / What over How)

摘要陳述程式碼達成之目標 ( What )，禁止逐行複述程式碼執行之操作步驟 ( How )。

- 摘要產出前，須先由程式碼之控制流程、資料流程、呼叫關係，推導其對應之業務意圖或功能目的，再以該意圖作為摘要之敘述主軸。
- 禁止使用程式語言關鍵字 ( 如迴圈、條件式、變數指派、回傳 ) 作為摘要之主要敘述動詞，除非該操作本身即為使用者可觀察之行為。
- 摘要句子以動作目的作為主詞與動詞核心 ( 例如：「驗證使用者身分」，而非「呼叫 checkAuth 函式並回傳布林值」)。
- 條件分支、迴圈邊界、例外捕捉等控制結構，僅於其直接決定業務結果時才於摘要中呈現，否則收斂至上層意圖之中。

理由：程式理解歷程存在由上而下 ( 假設驅動 ) 與由下而上 ( 語法組合 ) 兩種路徑；意圖提升原則將摘要固定於由上而下路徑之最終產出，避免摘要淪為程式碼之逐句同義詞替換。

### II. 去噪與抽象化 (De-noising and Abstraction)

摘要排除與核心邏輯無關之雜訊，並將重複或低階細節收斂為單一抽象敘述。

- 摘要排除下列項目：例外處理之樣板程式碼、日誌記錄陳述、型別轉換、匯入陳述、框架強制要求之樣板方法。
- 相似程式碼區塊重複出現三次以上者 ( 迴圈、遞迴、映射操作 )，摘要為單一敘述，禁止逐一列舉各次疊代之細節。
- 實作細節 ( 特定演算法選擇、暫存變數命名、內部資料結構 ) 收斂為其抽象角色 ( 「暫存結果」、「索引計數器」)，除非該細節為理解程式碼行為之必要條件。
- 巢狀層級超過使用者定義閾值之程式碼結構，摘要為其最外層之抽象目的，內層細節僅於語意分塊原則要求展開時才呈現。

理由：軟體逆向工程之抽象化程序，將實作層逐層提升至設計層與需求層；去噪與抽象化原則移植此程序，於摘要中保留語意訊號、剔除實作雜訊。

### III. 語意分塊 (Semantic Chunking)

摘要對象以語意完整之單元切分，切分邊界依循程式碼之自然結構。

- 切分邊界依循函式、類別、模組或內聚之程式碼區塊，禁止以固定行數或固定字元數切分。
- 不具備獨立呼叫或重用價值之私有輔助邏輯，併入其呼叫者之摘要範圍，不個別產出摘要。
- 跨越多個檔案而共同完成單一功能之程式碼 ( 控制器、服務層、資料存取層之呼叫鏈 )，辨識其呼叫鏈後產出涵蓋整體流程之摘要，並標註各分塊於流程中之角色。
- 每一語意區塊之摘要可獨立閱讀，不依賴讀者同時展開其他區塊之摘要即可理解其目的。

理由：摘要品質與輸入單元之語意內聚性相關；以語法邊界切分使摘要與程式碼之抽象單元錯位，語意分塊原則以函式、類別、模組等自然邊界取代語法邊界。

### IV. 領域詞彙映射 (Domain Vocabulary Mapping)

摘要將程式碼中之技術命名映射至所屬領域之業務詞彙。

- 摘要禁止直接沿用程式碼識別字 ( 變數名、函式名、類別名 ) 作為摘要用詞，除非該識別字本身已是領域詞彙。
- 領域詞彙之映射，以程式碼所屬專案之既有詞彙表、註解、文件、或測試案例命名作為依據，禁止臆測未於程式碼上下文中出現之業務語意。
- 程式碼識別字與領域詞彙產生衝突或歧義者 ( 縮寫、多重意義之命名 )，摘要標註為待釐清事項，不自行選定單一解釋。
- 摘要中使用之領域詞彙，於同一文件內保持一致，禁止對同一概念使用多個不同詞彙交替描述。

理由：軟體逆向工程由實作層還原至需求層時，須建立程式識別字與領域概念之對應表；領域詞彙映射原則要求摘要輸出使用領域語言，而非程式語言之字面翻譯，以符合需求方、維運人員、新進開發者之認知框架。

## 理論基礎 (Theoretical Foundations)

### 程式理解 (Program Comprehension)

程式理解為開發者於缺乏或不完整文件之情況下，重建程式碼運作機制與設計意圖之認知歷程。理解模型區分由上而下 ( 假設驅動，自功能目標尋找驗證證據 ) 與由下而上 ( 自陳述式逐步組合出高階結構 ) 兩種路徑。摘要產出程序模擬由上而下路徑之最終產出：以假設 ( 功能目的 ) 作為摘要起點，再以程式碼證據 ( 控制流程、資料流程 ) 驗證假設成立。

### 軟體逆向工程 (Software Reverse Engineering)

軟體逆向工程為於無原始設計文件之前提下，由既有實作反推系統結構、元件關係與需求規格之程序，區分三個抽象層級：實作層 ( 原始碼 )、設計層 ( 結構與元件關係 )、需求層 ( 系統目的與規格 )。摘要文件對應至需求層與設計層之間之產出物，須避免停留於實作層之逐句轉譯。

### 程式碼摘要 (Source Code Summarization)

程式碼摘要為將原始碼轉換為簡潔且可讀之自然語言敘述之產出程序，目的在於降低讀者建立心智模型所需之認知負荷。摘要品質評估面向包含：內容選擇 ( 是否涵蓋關鍵行為 )、抽象層級 ( 是否脫離逐行轉譯 )、語言精簡度 ( 是否排除冗餘敘述 )、讀者適配度 ( 是否對應讀者之詞彙體系 )。

## 原則彙整與對應 (Principle Consolidation and Mapping)

四項核心原則對應理論基礎之各面向，並聯合構成單一摘要產出程序：

- 意圖提升對應程式理解之由上而下路徑，決定摘要之抽象層級。
- 去噪與抽象化對應軟體逆向工程之實作層至設計層轉換，決定摘要之內容選擇與語言精簡度。
- 語意分塊對應程式碼摘要之輸入單元界定，決定摘要之切分邊界。
- 領域詞彙映射對應軟體逆向工程之設計層至需求層轉換，決定摘要之讀者適配度。

四項原則須聯合套用於單一摘要產出程序；任一原則之違反，視為摘要產出物不合規。

## Governance

- 修訂程序：原則之新增、刪除或重新定義，須提出修訂草案，並標示異動之原則編號與差異內容。
- 版本控制政策依循語意化版本規則：主要版本變更對應治理規則或原則之不相容變更；次要版本變更對應新增原則或章節、或既有原則之大幅擴充；補丁版本變更對應文字澄清、錯字修正、非語意性修訂。
- 合規性審查：摘要產出物於發佈前，須逐項核對本憲章四項核心原則，未通過核對之摘要不得視為最終產出。
