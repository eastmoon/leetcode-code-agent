# Pseudocode Constitution

## Core Principles

### I. 語言無關性 (Language Agnostic)
虛擬碼不得使用任何特定程式語言的關鍵字、語法糖或函式庫 API。
- 禁止出現語言專屬輸出/輸入函式 (例如 `printf`、`System.out.println`、`console.log`)。
- 禁止出現語言專屬模組匯入語法 (例如 `import`、`#include`、`require`)。
- 邏輯與控制結構以中立、通用的詞彙表示 (例如：輸入、輸出、迴圈、若...則...否則、回傳)。
- 資料型別以概念性名稱表示 (例如：整數、字串、清單、集合、對映)，不得使用特定語言的型別系統 (例如 `int32`、`std::vector`、`ArrayList`、`Dict[str, int]`)。
- 禁止使用語言專屬語法標記 (例如分號、大括號、縮排區塊字元)；區塊邊界以縮排與通用關鍵字表示。
- 理由：虛擬碼作為邏輯設計的中介表示，須可被任意程式語言實作，特定語言語法將限制其可轉譯性與可讀性。

### II. 結構清晰，善用縮進 (Structural Clarity via Indentation)
每一層邏輯區塊必須以縮排呈現其從屬關係。
- 迴圈、條件分支、函式主體等區塊必須以一致的縮排層級表示從屬結構，不得以其他符號取代縮排作為區塊邊界的主要依據。
- 同一份虛擬碼文件內縮排單位必須保持一致 (例如統一使用 2 個空白或 4 個空白)，不得混用不同縮排寬度。
- 巢狀結構的深度必須透過縮排逐層遞增呈現，每增加一層邏輯從屬即增加一層縮排。
- 關鍵控制字 (例如 若、否則、迴圈、結束) 必須置於其對應區塊縮排的起始或結束位置，明確標示區塊範圍。
- 理由：縮排是呈現邏輯層次最直觀的方式，一致的縮排規則可避免區塊歸屬產生歧義。

### III. 高度抽象化，隱藏實作細節 (High-Level Abstraction)
虛擬碼必須描述「做什麼」而非「如何實作」。
- 省略記憶體管理、指標運算、暫存器操作、底層系統呼叫等實作細節。
- 複雜操作以具語意的單一步驟或函式呼叫表示 (例如 `排序(清單)`、`查詢資料庫(條件)`)，不展開其內部演算法或程式庫實作細節。
- 錯誤與例外狀況以抽象概念表示 (例如 若發生錯誤則...)，不得描述特定語言的例外處理語法細節 (例如 try/catch/finally 之語言實作差異)。
- 僅展開屬於問題核心邏輯的步驟；輔助性、環境相依性操作 (例如記憶體釋放、型別轉換、連線逾時重試機制) 予以省略。
- 理由：虛擬碼的目的為傳達解題邏輯，過度描述實作細節將降低可讀性並使其與特定執行環境產生耦合。

### IV. 命名具備語意化 (Semantic Naming)
所有識別字命名必須反映其用途或所代表的意義。
- 禁止使用無意義代稱 (例如 `a`、`b`、`tmp`、`data1`、`x`)。
- 函式命名以動詞或動詞片語表示其行為 (例如 計算總和、驗證輸入、取得使用者資料)。
- 變數命名以名詞或名詞片語表示其所代表的資料 (例如 使用者清單、最大值、剩餘容量)。
- 布林變數或旗標命名須能表達真偽語意 (例如 是否已完成、是否有效、是否存在)。
- 同一份虛擬碼文件內命名風格必須保持一致 (例如統一使用底線分隔或駝峰式)，不得混用多種命名風格。
- 理由：語意化命名使虛擬碼無需額外注解即可被理解，降低轉譯為實際程式碼時產生誤解的風險。

## Governance

+ 修訂程序：任何原則的新增、刪除或修改，須先提出修訂提案並說明變更理由，經核准後方可更新本文件版本。
+ 版本控制政策：遵循語意化版本 (MAJOR.MINOR.PATCH)。
  - 主要版本：治理規則或核心原則發生不相容或重新定義之變更。
  - 次要版本：新增原則、新增章節，或對既有原則進行大幅擴充。
  - 補丁版本：文字澄清、措辭調整、拼字修正等非語意變更。
+ 合規性審查：所有虛擬碼撰寫產出須逐條對照本憲章四項核心原則檢查，不符合者須修正後方可視為完成。
