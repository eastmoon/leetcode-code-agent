# JavaScript Guideline

JavaScript 為動態型別、直譯執行（可搭配 JIT 編譯最佳化）之高階指令碼語言，具備原型鏈繼承（Prototypal Inheritance）、一級函式（First-Class Functions）、事件驅動與非同步（Asynchronous, Event Loop）執行模型等特性。本指南依循 `.codekit/memory/constitution/guideline.md` 與 `.codekit/memory/constitution/design.md`，將其核心原則詮釋為 JavaScript 語言之具體撰寫規範。

## Coding Rules

### I. 有意義的命名

- 類別與建構函式（Constructor Function）命名採名詞或名詞片語並以 PascalCase（如 `OrderProcessor`）；函式、方法與變數命名採 camelCase 且以動詞開頭（如 `calculateTotal`）。
- 禁止使用單一字母、縮寫或無意義代稱（如 `data`、`tmp`）命名變數、函式與類別；陣列迭代之慣用索引變數（`i`、`j`、`k`）與 `catch` 子句中之錯誤物件（`err`、`e`）除外。
- 全域不變常數以 `const` 宣告並採 SCREAMING_SNAKE_CASE（如 `MAX_RETRY_COUNT`）；同一概念於整個程式碼庫須使用一致之識別字，禁止對相同語意混用不同詞彙（如同時以 `get` 與 `fetch` 表示相同操作）。

**理由**：JavaScript 為動態型別語言，缺乏編譯期型別檢查輔助理解，命名品質直接決定程式碼可讀性與可維護性。

### II. 小型且單一抽象層級之函式

- 函式長度超過 30 行須拆分，依循單一職責原則評估拆分邊界。
- 函式參數個數超過 3 個須以物件解構（Object Destructuring）取代，禁止使用過長之位置參數列表。
- if/for 巢狀層級超過 2 層須以衛語句（Guard Clause）、提前回傳（Early Return）或抽取子函式重構；優先使用陣列方法（`map`、`filter`、`reduce`）取代手寫迴圈以降低巢狀層級。
- 單一函式僅維持單一抽象層級，禁止將高階流程控制與低階實作細節（如直接操作 DOM 或原始資料結構）混雜於同一函式。

### III. 自我說明程式碼優先於註解

- 程式碼須以命名、函式結構與型別註記（TypeScript 型別或 JSDoc）表達意圖；註解僅用於解釋「為何」而非「做什麼」。
- 公開 API 之語意須以清晰命名表達，必要時以 JSDoc（`/** */`）註解說明參數、回傳值與使用限制，禁止以註解彌補命名不良或結構混亂之程式碼。
- 禁止保留以 `//` 或區塊註解（`/* */`）註解掉之程式碼片段、過時註解或與實作行為不一致之註解。

### IV. 禁止重複

- 相同邏輯出現兩次以上須抽取為共用函式、高階函式（Higher-Order Function）或混入（Mixin），禁止複製貼上相同邏輯區塊。
- 常數與業務規則須以 `const` 集中定義於單一模組並以 `export` 匯出，作為單一權威來源，禁止於多處硬編碼相同數值（Magic Number）或字串字面值。

### V. 快速失敗與顯式錯誤處理

- 偵測到不合法狀態或非預期輸入，須立即以 `throw new Error(...)`（或自訂 `Error` 子類別）中止當前操作，禁止靜默忽略例外或吞噬 `Promise` 之 rejection。
- 非同步流程須統一使用 `async`／`await` 搭配 `try/catch` 處理錯誤，禁止於同一模組內混用回呼（Callback）風格與 `Promise` 鏈造成錯誤處理路徑分歧。
- 禁止回傳 `null`、`undefined` 或空物件掩蓋錯誤；函式應拋出明確例外，或回傳具明確語意之 Result 結構（如 `{ ok, value, error }`）。
- 禁止使用空的 `catch (error) {}` 區塊吞噬例外而不處理、記錄或重新拋出。

**理由**：JavaScript 之非同步例外若未被 `catch` 或 `.catch()` 捕捉，將以未處理拒絕（Unhandled Rejection）形式靜默失敗或造成程序終止，須以顯式錯誤處理策略防止錯誤狀態擴散。

### VI. 邊界輸入驗證

- 所有外部輸入（使用者輸入、HTTP 請求內容、第三方 API 回應、環境變數）進入函式邊界時，須以明確之型別與格式檢查（或 Schema 驗證函式庫）完成驗證。
- 驗證失敗須拒絕處理並拋出明確例外或回傳錯誤，禁止信任任何未經驗證之外部資料；禁止直接以未驗證之外部輸入拼接於查詢字串或以 `eval`／`new Function` 執行。
- 已於邊界完成驗證之資料在內部模組間傳遞時，禁止重複進行相同驗證造成邏輯冗贅。

### VII. 斷言與前置條件

- 函式之前置條件、後置條件、不變量須以顯式檢查陳述式（`if (!condition) throw new Error(...)`）明確表達；開發環境除錯得輔以 `console.assert`。
- 斷言僅用於偵測程式錯誤（Bug），禁止用於處理可預期的執行期錯誤（如使用者輸入錯誤），此類情境須以第 VI 條之邊界驗證機制處理。
- 正式環境（Production）建置不得僅依賴 `console.assert` 作為防護手段，關鍵不變量須轉換為顯式錯誤拋出，確保 Production 建置中仍保留防護。

### VIII. 不變性與最小化副作用

- 資料結構預設以 `const` 宣告並優先使用不可變操作（展開運算子 `...`、`Array.prototype.map`／`filter`、`Object.freeze`）取代直接變異（`push`、`splice`、屬性直接賦值）。
- 函式須避免修改傳入之參數物件或全域狀態；必要之副作用（I/O、DOM 操作、全域變數修改）須於函式命名（如 `mutateXxx`、`applyXxx`）明確標示。
- 共享狀態（模組層級變數、全域物件）之存取須最小化，降低事件迴圈（Event Loop）中非同步回呼交錯執行時之競態風險。

### IX. 一致的格式與結構

- 專案須採用單一 ESLint 與 Prettier 設定檔案並以 CI 強制執行，禁止手動調整縮排、換行與括號風格；相關程式碼須依邏輯或呼叫順序排列。
- 模組須統一採用 ES Module（`import`／`export`）語法，禁止於同一專案內混用 CommonJS（`require`）與 ES Module。
- 每個檔案僅公開必要之匯出（Named Export 或單一 Default Export），內部輔助函式須維持模組私有（不匯出）。

### X. 童子軍原則

- 每次修改既有 JavaScript 檔案時，須讓其比修改前更清晰整潔，發現 `var` 宣告、隱式全域變數、未處理之 `Promise` 或命名不良之程式碼，須於能力範圍內即時修正（如將 `var` 改為 `const`／`let`）。
- 技術債務（如暫時性 `TODO`、過時之相容性程式碼、已棄用之 API 呼叫）須明確記錄，不得無限期擱置或於既有混亂基礎上疊加新的混亂。

## Design Constraints

### I. 單一職責原則

- 每個類別、模組或函式僅維持一個引發修改之原因；公開方法或匯出函式數量超出職責範圍須評估拆分為獨立模組。
- 建構函式（`constructor`）僅負責初始化物件狀態，禁止於建構函式內執行業務邏輯運算或發起非同步請求。

### II. 開放封閉原則

- 既有已驗證之類別與模組行為不得直接修改；新行為須透過繼承（`extends`）、組合（Composition）或以函式參數傳入之策略模式擴充。
- 以 `Object.freeze` 或私有欄位（`#field`）封閉不預期外部修改之內部狀態與類別結構。

### III. 里氏替換原則

- 子類別覆寫父類別方法時，不得強化前置條件（要求更嚴格之參數）或削弱後置條件（回傳與父類別契約不一致之型別或結構）。
- 子類別方法簽章（參數個數與型別）須與父類別宣告一致，呼叫端以父類別介面呼叫子類別實例時行為須保持正確。

### IV. 介面隔離原則

- 以物件字面值或抽象基底類別定義之介面須依使用方需求拆分；單一介面方法數超過 5 個須評估拆分為多個角色介面（如以多個小型 Mixin 組合）。
- 禁止要求實作方（實作特定合約之物件或類別）實作與其職責無關之方法。

### V. 依賴反轉原則

- 高層模組須依賴抽象（函式參數注入之依賴、介面形狀約定），禁止於業務邏輯內部直接以具體實作模組路徑 `import` 並實例化低層模組。
- 依賴注入須透過函式參數、建構函式參數或工廠函式（Factory Function）達成，便於測試時以替身（Test Double）取代真實依賴。

### VI. 高內聚低耦合

- 模組須依單一明確功能群組劃分檔案，相關函式與類別集中於同一模組內。
- 模組間依賴須透過明確之 `export` 介面暴露，禁止跨模組直接存取另一模組之內部（未匯出）變數或函式。
- 新增功能時，須優先評估是否歸屬於現有模組，而非建立新的跨模組依賴路徑。

### VII. 關注點分離

- 資料存取（如 API 呼叫、資料庫存取）、業務邏輯、呈現層（UI 元件、視圖渲染）須拆分為獨立之模組或檔案，禁止於同一檔案內混合三者邏輯。
- 跨切面關注點（日誌、快取、身分驗證）須以中介層（Middleware）或高階函式（Higher-Order Function）包裝實作，禁止散佈於業務邏輯程式碼各處。
