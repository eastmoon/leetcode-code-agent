# Python Guideline

Python 為直譯式、動態型別、支援多典範（物件導向、函式式、程序式）之程式語言，具備 PEP 8 風格規範、型別提示（Type Hints）、例外處理（Exception Handling）機制與標準函式庫。

## Coding Rules

### I. Meaningful Naming (有意義的命名)

- 函式與變數命名須使用 `snake_case`，並揭示意圖，禁止使用單字母、縮寫或無意義代稱（如 `data`、`tmp`、`x1`）。
- 類別（class）命名須使用 `PascalCase`；常數命名須使用 `UPPER_SNAKE_CASE`；模組（module）與套件（package）名稱須使用小寫 `snake_case`。
- 類別命名為名詞或名詞片語；函式命名為動詞或動詞片語，準確反映其行為。
- 同一概念於整個程式碼庫須使用一致詞彙，禁止對同一事物使用多個不同名稱（如同時使用 `get_`、`fetch_`、`retrieve_` 表示相同行為）。

**理由**：清晰命名降低理解成本，減少因命名歧義導致的誤用與缺陷。

---

### II. Small, Single-Purpose Functions (小型單一目的函式)

- 函式須只做一件事，並將該件事做好；函式內混合多層抽象須拆分為輔助函式（helper function）。
- 參數個數超過 3 個須以 `dataclass`、`TypedDict` 或關鍵字引數物件封裝取代。
- 巢狀層級（if/for 巢狀）超過 2 層須重構為衛語句（Guard Clause，提前 `return`/`continue`）或抽取子函式。

**理由**：小型函式易於理解、測試與重用，降低認知負擔。

---

### III. Self-Documenting Code Over Comments (自我說明優先於註解)

- 程式碼須透過命名與結構表達意圖；文件字串（docstring，`"""..."""`）與行內註解（`#`）僅用於解釋「為何」而非「做什麼」。
- 禁止保留註解掉的程式碼（commented-out code）、過時註解或與實作行為不一致的註解。
- 禁止使用註解彌補命名不良或結構混亂的程式碼。

**理由**：程式碼是唯一不會過期的文件；註解易與實作脫節，造成誤導。

---

### IV. No Duplication (DRY, Don't Repeat Yourself)

- 相同邏輯禁止複製貼上；重複出現兩次以上的邏輯須抽取為共用函式或共用模組。
- 常數、列舉（`enum.Enum`）、業務規則須有單一權威來源（Single Source of Truth），禁止於多處硬編碼相同字面值。

**理由**：重複邏輯造成維護時多處修改風險，容易產生不一致缺陷。

---

### V. Fail Fast & Explicit Error Handling (快速失敗與顯式錯誤處理)

- 偵測到不合法狀態或非預期輸入，須立即以 `raise` 拋出例外（如 `ValueError`、`TypeError`），禁止以空的 `except:` 或 `except Exception: pass` 靜默吞噬例外。
- 禁止回傳 `None` 掩蓋錯誤；須以明確例外或 `Optional[T]` 型別提示（Type Hint）搭配呼叫端顯式檢查表達可能缺值狀態。
- 例外（Exception）僅用於例外狀況；當條件可由 `if` 陳述式直接判斷時（如檢查鍵是否存在、索引是否越界），禁止以 `try/except` 取代明確條件判斷。

**理由**：儘早暴露錯誤可縮小除錯範圍，避免錯誤狀態擴散至系統其他部分。

---

### VI. Input Validation at Boundaries (邊界輸入驗證)

- 所有外部輸入（命令列參數、API 請求、檔案內容、環境變數）進入函式或模組邊界時，須完成型別、範圍、格式驗證。
- 邊界函式須以型別提示（Type Hint）搭配明確條件判斷或驗證函式庫（如 `pydantic`）拒絕不合法輸入，禁止信任任何未經驗證之外部資料。
- 已於邊界驗證之資料在內部函式間傳遞時，禁止重複進行相同驗證造成邏輯冗贅。

**理由**：邊界驗證阻絕不可信資料流入核心邏輯，降低注入與崩潰風險。

---

### VII. Assertions & Preconditions (斷言與前置條件)

- 函式對於前置條件、後置條件、不變量須以 `assert` 陳述式明確表達。
- `assert` 僅用於偵測程式錯誤（Bug），禁止用於處理可預期的執行期錯誤（如使用者輸入錯誤）；直譯器以 `-O` 旗標執行時 `assert` 會被移除，關鍵檢查須改以明確 `raise` 例外實作，不得完全依賴 `assert`。

**理由**：斷言將隱性假設轉換為可執行的檢查，使違反假設的狀態立即可見。

---

### VIII. Immutability & Minimized Side Effects (不變性與最小化副作用)

- 資料結構優先使用 `tuple`、`frozenset` 或 `@dataclass(frozen=True)`，取代可變（mutable）結構。
- 函式參數禁止使用可變預設值（如 `def f(items=[])`），須改用 `None` 搭配函式內初始化。
- 函式須避免修改輸入參數或模組層級全域變數；必要之副作用須於函式命名或文件字串明確標示。
- 共享狀態的存取須以 `threading.Lock` 或並行安全結構最小化，降低多執行緒環境下的競態風險。

**理由**：不變性與最小副作用降低狀態追蹤複雜度，提升可測試性與並行安全性。

---

### IX. Consistent Formatting & Structure (一致的格式與結構)

- 專案須遵循 PEP 8 格式規範，並以自動化工具（如 `black`、`ruff`、`flake8`）強制執行縮排、換行與命名慣例。
- `import` 陳述式須依標準函式庫、第三方套件、本地模組分組排序（如以 `isort` 強制執行），並移除未使用之 `import`。
- 模組與套件結構須遵循一致慣例，禁止同一專案內混用不同組織風格。

**理由**：一致格式降低視覺解析成本，使程式碼審查聚焦於邏輯而非風格差異。

---

### X. Boy Scout Rule (童子軍原則)

- 每次修改程式碼時，須讓其比修改前更清晰整潔，禁止在既有混亂基礎上疊加新的混亂。
- 發現技術債務（Technical Debt）須以 `# TODO:` 標記並記錄追蹤事項，或於能力範圍內即時修正，不得無限期擱置。

**理由**：持續性微幅改善避免程式碼腐化累積至無法維護的臨界點。

## Design Constraints

### I. Single Responsibility Principle (SRP)

- 每個類別或函式只能有一個引發變更的原因。
- 當一個類別同時承擔多於一項職責時，須立即拆分為獨立類別或模組。
- 函式本體超過 30 行，須檢視是否違反單一職責並評估拆分。

**理由**：單一職責使變更範圍可控，避免修改一處邏輯連帶影響無關功能。

---

### II. Open/Closed Principle (OCP)

- 既有模組對擴展開放、對修改封閉；新需求須透過抽象基底類別（`abc.ABC`）、組合（Composition）或策略模式（Strategy Pattern）擴展，不得直接修改已穩定的核心邏輯。
- 禁止以 monkey-patching 修改已穩定第三方模組或核心模組之行為。
- 任何修改既有已驗證邏輯行為的變更，須附上充分理由與影響評估。

**理由**：降低修改已驗證程式碼所帶來的回歸風險，使系統可持續擴展。

---

### III. Liskov Substitution Principle (LSP)

- 子類別須可完整替代父類別使用，且不改變程式正確性。
- 子類別覆寫方法時，禁止削弱父類別的前置條件，亦不得強化後置條件。
- 若繼承導致行為不一致或違反父類別契約，須以組合（Composition）取代繼承。

**理由**：確保多型行為的一致性，防止呼叫端因替換子類別而產生隱性執行期錯誤。

---

### IV. Interface Segregation Principle (ISP)

- 抽象介面（`typing.Protocol` 或 `abc.ABC`）須依使用方需求拆分；禁止建立大型通用介面，強迫實作方實作不需要的方法。
- 當介面方法數超過 5 個時，須評估是否應拆分為多個角色協定（Protocol）。

**理由**：細粒度介面使依賴關係精確，降低不必要的耦合。

---

### V. Dependency Inversion Principle (DIP)

- 高層模組不得直接依賴低層模組的具體實作；高層與低層模組均須依賴抽象（`abc.ABC` 或 `typing.Protocol`）。
- 依賴注入（透過建構子或函式參數傳入依賴）為標準實踐；禁止在業務邏輯內部直接實例化具體依賴。
- 抽象層定義於高層模組所在之套件，由低層模組實作該抽象。

**理由**：依賴反轉使高層邏輯可在不修改的情況下替換底層實作，提升可測試性與可替換性。

---

### VI. High Cohesion, Low Coupling (高內聚低耦合)

- 每個模組或類別僅負責單一明確的功能群組，相關邏輯須集中於同一單元內。
- 模組間的依賴關係須最小化；跨模組溝通僅能透過明確定義的公開介面進行。
- 內部實作細節須以 `_leading_underscore` 命名慣例標示為私有，禁止外部模組直接存取。
- 新增功能時，須優先評估是否歸屬於現有模組，而非建立新的跨模組依賴路徑。

**理由**：高內聚確保變更影響範圍可預測；低耦合確保模組可獨立替換、測試與部署。

---

### VII. Separation of Concerns (SoC)

- 資料存取、業務邏輯、呈現層（如 CLI、Web API）須實體分離於不同模組或套件，不得混合於同一檔案。
- 每個層次僅允許向相鄰層傳遞資料，禁止跨層直接呼叫（如呈現層直接執行資料庫查詢）。
- 設定（Configuration）、路由（Routing）、驗證（Validation）、日誌（Logging）各自獨立管理，不得嵌入業務邏輯中。
- 跨切面關注點（Logging、Security、Caching）須以裝飾器（Decorator）或中介層（Middleware）實作，禁止散佈於業務程式碼各處。

**理由**：關注點分離使各層可獨立演進，降低連鎖修改風險，提升可測試性與可維護性。

## Governance

- 本指南凌駕於所有其他 Python 開發慣例與口頭約定之上，適用範圍涵蓋所有以 Python 撰寫之程式碼。
- 指南修訂須記錄變更原因、影響範圍及遷移計畫，並由專案維護者核准後方可生效。
- 版本控制遵循語意版本規則：
  - 主要版本（MAJOR）：原則重新定義或不相容變更。
  - 次要版本（MINOR）：新增原則或章節，或大幅擴展具體指導方針。
  - 補丁版本（PATCH）：措詞澄清、錯字修正、非語意改進。
- 所有程式碼審查（Code Review）須逐一驗證上述 Coding Rules 與 Design Constraints 之遵守情況；違反須於 PR 描述中列出具體理由並獲得維護者核准方可合併。
- 合規性審查於每次里程碑（Milestone）結束時執行。
