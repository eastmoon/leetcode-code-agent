# C++ Guideline

C++ 為靜態型別、編譯至原生碼之通用程式語言，具備手動記憶體管理、範本後設程式設計（Template Metaprogramming）與零額外開銷抽象（Zero-Overhead Abstraction）特性。本指南依循 `.codekit/memory/constitution/guideline.md` 與 `.codekit/memory/constitution/design.md`，將其核心原則詮釋為 C++ 語言之具體撰寫規範。

## Coding Rules

### I. 有意義的命名

- 類別與結構命名採名詞或名詞片語（如 `OrderProcessor`）；函式與方法命名採動詞開頭（如 `calculateTotal`）。
- 禁止使用單一字母、縮寫或無意義代稱（如 `data`、`tmp`）命名變數、類別與函式；迴圈索引變數（`i`、`j`、`k`）除外。
- 同一概念於整個程式碼庫須使用一致之識別字，禁止對相同語意混用不同詞彙（如同時以 `Get` 與 `Fetch` 表示相同操作）。

**理由**：C++ 缺乏語言層級之強制文件字串機制，命名品質直接決定標頭檔介面之可讀性與可維護性。

### II. 小型且單一抽象層級之函式

- 函式長度超過 30 行須拆分，依循單一職責原則評估拆分邊界。
- 函式參數個數超過 3 個須以 `struct` 或類別封裝取代，禁止使用過長之參數列表。
- if/for 巢狀層級超過 2 層須以衛語句（Guard Clause）、提前回傳（Early Return）或抽取子函式重構。
- 單一函式僅維持單一抽象層級，禁止將高階流程控制與低階實作細節混雜於同一函式。

### III. 自我說明程式碼優先於註解

- 程式碼須以型別、命名與函式結構表達意圖；註解僅用於解釋「為何」而非「做什麼」。
- 公開介面（標頭檔宣告）之語意須以清晰命名與型別簽章表達，必要時以 Doxygen 風格（`///`）註解說明使用限制，禁止以註解彌補命名不良或結構混亂之程式碼。
- 禁止保留以 `//` 或 `#if 0` 註解掉之程式碼片段、過時註解或與實作行為不一致之註解。

### IV. 禁止重複

- 相同邏輯出現兩次以上須抽取為共用函式、範本函式（Template Function）或基底類別方法，禁止複製貼上相同邏輯區塊。
- 常數與業務規則須以 `constexpr` 或 `enum class` 集中定義於單一標頭檔，作為單一權威來源，禁止於多處硬編碼相同數值（Magic Number）。

### V. RAII 資源生命週期與快速失敗之錯誤處理

- 所有取得之資源（記憶體、檔案控標、鎖、Socket）須封裝於物件之建構子與解構子中管理生命週期；禁止以裸指標搭配 `new`／`delete` 管理物件所有權，改用 `std::unique_ptr`、`std::shared_ptr` 或標準容器。
- 解構子須標記 `noexcept`，並保證例外拋出路徑上資源仍可正確釋放。
- 同一模組內錯誤處理策略須一致，禁止混用例外（Exception）與錯誤碼表示同一類錯誤情境；例外僅用於不可預期之錯誤情境，可預期之失敗須以 `std::optional`、`std::expected` 表示。
- 禁止以空的 `catch (...)` 區塊吞噬例外而不處理或重新拋出。

**理由**：C++ 無自動垃圾回收機制，資源生命週期須由物件生命週期繫結；同時錯誤須即刻暴露而非被吞噬或以無意義預設值掩蓋，兩者共同構成 C++ 之例外安全（Exception Safety）基礎。

### VI. 邊界輸入驗證

- 所有外部輸入（使用者輸入、檔案內容、網路封包、環境變數、第三方 API 回應）進入函式邊界時，須以型別安全之解析方式（如 `std::from_chars`、`std::stoi` 搭配例外捕捉）完成型別、範圍與格式驗證。
- 驗證失敗須拒絕處理並以例外或 `std::expected` 回報明確錯誤，禁止信任任何未經驗證之外部資料。
- 已於邊界完成驗證之資料在內部模組間傳遞時，禁止重複進行相同驗證造成邏輯冗贅。

### VII. 斷言與前置條件

- 函式之前置條件、後置條件、不變量須以 `assert`（執行期、可於 Release 建置停用）或 `static_assert`（編譯期）明確表達。
- 斷言僅用於偵測程式錯誤（Bug），禁止用於處理可預期的執行期錯誤（如使用者輸入錯誤），此類情境須以第 VI 條之邊界驗證機制處理。
- 具重要回傳值之函式須標記 `[[nodiscard]]`，避免呼叫端忽略必須處理之結果或錯誤碼。

### VIII. Const 正確性與現代型別安全

- 不修改物件狀態之成員函式須標記為 `const`；不重新指派之區域變數與參數須以 `const` 或 `constexpr` 宣告。
- 傳遞唯讀之大型物件時，須以 `const&` 傳遞，禁止不必要之值複製；函式須避免修改輸入參數或全域狀態。
- 型別轉換須使用具名轉換運算子（`static_cast`、`dynamic_cast`、`const_cast`），禁止使用 C 風格轉型。
- 列舉型別須使用 `enum class`，禁止使用不具作用域之 `enum`；禁止以巨集（`#define`）定義常數或函式，改用 `constexpr` 變數與 `inline` 函式。

### IX. 一致的格式與結構、標頭檔衛生

- 專案須採用單一 `.clang-format` 設定檔案並以 CI 強制執行，禁止手動調整縮排、換行與括號風格；相關程式碼須依邏輯或呼叫順序排列。
- 標頭檔須使用 `#pragma once` 或包含防護（Include Guard）防止重複引入；標頭檔內禁止使用 `using namespace` 陳述式。
- 標頭檔僅公開必要之介面，實作細節須置於原始檔或以 PIMPL 慣用法隱藏，降低編譯依賴。

### X. 童子軍原則

- 每次修改既有 C++ 檔案時，須讓其比修改前更清晰整潔，發現裸指標所有權不明確、資源洩漏風險或命名不良之程式碼，須於能力範圍內即時修正。
- 技術債務（如暫時性 `TODO`、過時之相容性程式碼）須明確記錄，不得無限期擱置或於既有混亂基礎上疊加新的混亂。

## Design Constraints

### I. 單一職責原則

- 每個類別僅維持一個引發修改之原因；公開方法數量超出職責範圍須評估拆分。
- 建構子僅負責初始化物件狀態，禁止於建構子內執行業務邏輯運算。

### II. 開放封閉原則

- 既有已驗證之類別階層須以 `final` 明確封閉不預期擴充之類別與虛擬函式。
- 新行為須透過衍生類別、範本特化或策略模式擴充，禁止修改既有穩定基底類別之虛擬函式簽章。

### III. 里氏替換原則

- 衍生類別覆寫虛擬函式時，不得強化前置條件或削弱後置條件。
- 衍生類別之 `noexcept` 規格不得比基底類別更嚴格；覆寫函式簽章須與基底宣告一致。

### IV. 介面隔離原則

- 抽象基底類別（純虛擬類別）之純虛擬函式數量須維持精簡；超過 5 個須評估拆分為多個角色介面。
- 禁止要求衍生類別實作與其職責無關之虛擬函式。

### V. 依賴反轉原則

- 高層模組須依賴抽象基底類別或範本介面，禁止直接以 `new` 實例化低層模組之具體類別。
- 依賴注入須透過建構子參數（介面指標／參考）或範本參數（Policy-Based Design）達成。

### VI. 高內聚低耦合

- 類別與命名空間須依單一明確功能群組劃分；同一命名空間內型別須具備高度關聯性。
- 模組間依賴須透過抽象基底類別或函式介面暴露，禁止跨編譯單元直接存取另一模組之私有成員或全域變數。
- 標頭檔實作細節須以 PIMPL 慣用法隱藏，降低編譯依賴與耦合面。

### VII. 關注點分離

- 資料存取、業務邏輯、呈現層須拆分為獨立之類別或命名空間，禁止於同一類別內混合三者邏輯。
- 跨切面關注點（日誌、快取）須以裝飾器（Decorator）或範本方法（Template Method）等機制實作，禁止散佈於業務邏輯類別中。

## Governance

- 本指南依附於 `.codekit/memory/constitution/guideline.md` 與 `.codekit/memory/constitution/design.md`；與憲章牴觸時，以憲章為準。
- 指南修訂須記錄變更原因與影響範圍，並同步更新版本號與修訂日期。
- 版本控制遵循語意版本規則：主要版本為原則重新定義或不相容變更；次要版本為新增章節或大幅擴充細則；補丁版本為措詞澄清或錯字修正。
- 合規性審查須於每次程式碼審查（Code Review）時逐一核對本指南所列鐵律。
